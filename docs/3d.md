# 3D Rendering

> **Owner subagent:** `r3f-engineer`
> **Touches:** Anything inside `<Canvas>`, Three.js, React Three Fiber, drei, `useGLTF`, scene rendering, `WorldViewer` and related
> **Does NOT touch:** API routes, page chrome (header/footer/forms), DB

## Stack

- **Renderer:** React Three Fiber (R3F)
- **Helpers:** `@react-three/drei` (Bounds, Center, useGLTF, OrbitControls, etc.)
- **Engine:** Three.js (underlying)
- **Asset format:** `.glb` (binary glTF 2.0) — currently the entire world is one file
- **Phase 2 target:** scene graph composition — many `.glb` assets referenced from a JSON scene graph

## Current World Rendering

A world today = one `.glb` file. The viewer:

1. Loads the `.glb` via `useGLTF` (drei)
2. Wraps in `<Bounds>` (drei) to auto-fit the camera to the model's bounding box
3. Wraps in `<Center>` (drei) to recenter at origin
4. Renders inside a `<Canvas>` with OrbitControls + lighting + fullscreen toggle

### `<WorldViewer>` component

- File: `src/components/world-viewer/WorldViewer.tsx`
- Props: `glbUrl: string` (required), `ariaLabel?: string` (optional, defaults to `"3D world viewer"`)
- Lazy-loaded on `/world/[id]` page via `next/dynamic` with `ssr: false` in `src/app/world/[id]/WorldViewerClient.tsx`. Not imported in the feed — Three.js stays out of that bundle.
- Fullscreen toggle: a `<FullscreenButton>` component (defined in the same file) holds a `containerRef` to the wrapping `<div>`. On click it calls `containerRef.current.requestFullscreen()` to enter or `document.exitFullscreen()` to exit. It syncs button state by listening to the `fullscreenchange` DOM event. Errors (e.g., iframe restrictions) are caught and logged as a warning; the UI does not crash.

## Asset Loading Patterns

- `useGLTF` (drei) for the primary world model. The `<Model url={glbUrl}>` inner component calls `useGLTF` and suspends until the file is ready; it is wrapped in `<Suspense fallback={null}>` inside the Canvas.
- `useGLTF.preload(url)` is not called in the current implementation. It is documented as a future optimisation (trigger from the feed when navigation to a world is imminent).
- Error handling: `<WorldViewerErrorBoundary>` (class component — `src/components/world-viewer/WorldViewerErrorBoundary.tsx`) wraps the whole viewer. Any render error, including a failed GLB load, is caught and replaced with `<ViewerError>`, which shows "Couldn't load this world — The 3D file may be corrupted or unreachable. Try refreshing."
- Loading state UI: `<LoadingOverlay>` reads `useProgress().active` from drei (a zustand hook, callable outside the Canvas) and renders `<ViewerLoading>` — a semi-transparent backdrop with "Loading world…" text — while a load is in progress. Both fallback components live in `src/components/world-viewer/WorldViewerFallback.tsx`.

## Performance Budget

- **`.glb` size cap:** 50 MB (enforced in `POST /api/uploads/sign` via `KIND_RULES.glb.maxBytes = 50 * 1024 * 1024`). The 25 MB figure in early planning was not what shipped.
- **Polygon count:** no hard cap enforced. No specific threshold has been measured yet for lower-end devices; this is a future optimisation concern.
- **Texture size:** no explicit cap enforced on texture dimensions or total texture memory. The 50 MB GLB cap is the only hard constraint on the asset today.
- **Feed-card preview:** `preload="none"` on `<video>` (handled by `WorldCardMedia`, not 3D-rendered in the feed)

## Lighting Defaults

- Ambient light: `<ambientLight intensity={0.5} />` — no explicit color (defaults to white `#ffffff`)
- Directional light: `<directionalLight position={[5, 5, 5]} intensity={1} />` — no explicit color (white), no shadow casting configured
- HDRI environment: `<Environment preset="studio" />` from drei is used, loaded inside the `<Suspense>` block. It provides image-based lighting (IBL) on top of the two explicit lights. There is no hemisphere light in the current implementation.

## Camera Defaults

- Initial position: `[3, 3, 5]` (passed to `<Canvas camera={{ position: [3, 3, 5], fov: 50 }}>`)
- FOV: `50` degrees
- `<Bounds fit clip observe margin={1.4}>` overrides the initial position at mount to auto-fit the loaded model's bounding box. `margin={1.4}` adds 40% breathing room so the model is not flush against the frustum edges.
- OrbitControls: `enableDamping`, `dampingFactor={0.05}`, `minDistance={0.5}`, `maxDistance={50}`. No polar angle clamps are set (full vertical orbit allowed).

## Phase 2 Prep — Scene Graph Rendering

Phase 2 transforms a world from "one .glb file" to "a scene graph that composes many `.glb` assets in 3D space." This is the biggest change to the 3D layer in the entire roadmap. Key design considerations:

### Scene graph schema (v1)

A versioned JSON document. Minimal v1 (subject to refinement during Phase 2 planning):

```json
{
  "schemaVersion": 1,
  "objects": [
    {
      "id": "obj_abc",
      "assetId": "asset_123",
      "transform": {
        "position": [0, 0, 0],
        "rotation": [0, 0, 0],
        "scale": [1, 1, 1]
      },
      "name": "Tree"
    }
  ],
  "lights": [
    { "type": "sun", "intensity": 1.0, "direction": [...] },
    { "type": "ambient", "intensity": 0.3 }
  ],
  "environment": {
    "skybox": "sunset",
    "fog": null
  },
  "spawnPoints": [
    { "id": "default", "position": [0, 1.6, 5], "rotation": [0, 0, 0] }
  ],
  "camera": {
    "defaultPosition": [...],
    "defaultTarget": [...]
  }
}
```

### Renderer behavior

- If `worlds.scene_graph` is present → render via scene graph renderer (iterate objects, instantiate `.glb` per `assetId`, apply transform)
- Else → fall back to legacy single-`.glb` renderer (current behavior)
- One-click "convert legacy to scene graph": wraps the current `.glb` as a single-object scene graph

### Performance considerations for Phase 2

- **Instancing** — same `assetId` repeated many times should share GPU geometry (R3F `<Instances>` from drei)
- **Frustum culling** — Three.js does this by default; ensure it's not disabled
- **LOD** — likely not in v1, but design the asset model to allow LOD variants later
- **Async loading** — show partial scene as assets stream in

### Editor mode (Phase 2)

The in-browser editor reuses the renderer but adds:

- Transform gizmos (`<TransformControls>` from drei)
- Selection state (clicked object highlighted)
- Drag-to-place from asset panel
- Touch-friendly gizmos (tablets day one — touch handling is a real research item)

## Phase 3+ Prep

- **Shared presence:** capsule avatars rendered alongside the world. Simple — sphere + cylinder + name label sprite.
- **Realtime sync:** position/rotation updates from collaborators arrive via Liveblocks (or Yjs) and update avatar transforms.
- **Realtime editing:** scene graph mutations from other editors apply in real time. R3F's reactive rendering handles this for free if state is shared correctly.

## Gotchas

- **`useGLTF` caches by URL** — if you re-upload to the same URL, the cached version may persist. Force a refresh by appending a versioned query param (`?v=2`).
- **`<Suspense>` is required** around `useGLTF` consumers in R3F — provide a fallback or the canvas crashes on first load.
- **Pointer events on `<Canvas>`** — can interfere with page scrolling. Make sure the canvas doesn't capture wheel events when not focused.
- **R3F + Next.js App Router** — `<Canvas>` MUST be in a client component (`"use client"` directive at the top). Server-rendering R3F crashes.

## Slice 7 (3D-related)

Slice 7 has no 3D changes. View counts are tracked on the page, not in the 3D layer.
