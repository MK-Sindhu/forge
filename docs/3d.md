# 3D Rendering

> **Owner subagent:** `r3f-engineer`
> **Touches:** Anything inside `<Canvas>`, Three.js, React Three Fiber, drei, `useGLTF`, scene rendering, `WorldViewer` and related
> **Does NOT touch:** API routes, page chrome (header/footer/forms), DB

## Stack

- **Renderer:** React Three Fiber (R3F)
- **Helpers:** `@react-three/drei` (Bounds, Center, useGLTF, OrbitControls, etc.)
- **Engine:** Three.js (underlying)
- **Asset format:** `.glb` (binary glTF 2.0) â€” currently the entire world is one file
- **Phase 2 target:** scene graph composition â€” many `.glb` assets referenced from a JSON scene graph

## Current World Rendering

A world today = one `.glb` file. The viewer:

1. Loads the `.glb` via `useGLTF` (drei)
2. Wraps in `<Bounds>` (drei) to auto-fit the camera to the model's bounding box
3. Wraps in `<Center>` (drei) to recenter at origin
4. Renders inside a `<Canvas>` with OrbitControls + lighting + fullscreen toggle

### `<WorldViewer>` component

- File: `src/components/world-viewer/WorldViewer.tsx`
- Props: `glbUrl: string` (required), `ariaLabel?: string` (optional, defaults to `"3D world viewer"`)
- Lazy-loaded on `/world/[id]` page via `next/dynamic` with `ssr: false` in `src/app/world/[id]/WorldViewerClient.tsx`. Not imported in the feed â€” Three.js stays out of that bundle.
- Fullscreen toggle: a `<FullscreenButton>` component (defined in the same file) holds a `containerRef` to the wrapping `<div>`. On click it calls `containerRef.current.requestFullscreen()` to enter or `document.exitFullscreen()` to exit. It syncs button state by listening to the `fullscreenchange` DOM event. Errors (e.g., iframe restrictions) are caught and logged as a warning; the UI does not crash.

## Asset Loading Patterns

- `useGLTF` (drei) for the primary world model. The `<Model url={glbUrl}>` inner component calls `useGLTF` and suspends until the file is ready; it is wrapped in `<Suspense fallback={null}>` inside the Canvas.
- `useGLTF.preload(url)` is not called in the current implementation. It is documented as a future optimisation (trigger from the feed when navigation to a world is imminent).
- Error handling: `<WorldViewerErrorBoundary>` (class component â€” `src/components/world-viewer/WorldViewerErrorBoundary.tsx`) wraps the whole viewer. Any render error, including a failed GLB load, is caught and replaced with `<ViewerError>`, which shows "Couldn't load this world â€” The 3D file may be corrupted or unreachable. Try refreshing."
- Loading state UI: `<LoadingOverlay>` reads `useProgress().active` from drei (a zustand hook, callable outside the Canvas) and renders `<ViewerLoading>` â€” a semi-transparent backdrop with "Loading worldâ€¦" text â€” while a load is in progress. Both fallback components live in `src/components/world-viewer/WorldViewerFallback.tsx`.

## Performance Budget

- **`.glb` size cap:** 50 MB (enforced in `POST /api/uploads/sign` via `KIND_RULES.glb.maxBytes = 50 * 1024 * 1024`). The 25 MB figure in early planning was not what shipped.
- **Polygon count:** no hard cap enforced. No specific threshold has been measured yet for lower-end devices; this is a future optimisation concern.
- **Texture size:** no explicit cap enforced on texture dimensions or total texture memory. The 50 MB GLB cap is the only hard constraint on the asset today.
- **Feed-card preview:** `preload="none"` on `<video>` (handled by `WorldCardMedia`, not 3D-rendered in the feed)

## Lighting Defaults

- Ambient light: `<ambientLight intensity={0.5} />` â€” no explicit color (defaults to white `#ffffff`)
- Directional light: `<directionalLight position={[5, 5, 5]} intensity={1} />` â€” no explicit color (white), no shadow casting configured
- HDRI environment: `<Environment preset="studio" />` from drei is used, loaded inside the `<Suspense>` block. It provides image-based lighting (IBL) on top of the two explicit lights. There is no hemisphere light in the current implementation.

## Camera Defaults

- Initial position: `[3, 3, 5]` (passed to `<Canvas camera={{ position: [3, 3, 5], fov: 50 }}>`)
- FOV: `50` degrees
- `<Bounds fit clip observe margin={1.4}>` overrides the initial position at mount to auto-fit the loaded model's bounding box. `margin={1.4}` adds 40% breathing room so the model is not flush against the frustum edges.
- OrbitControls: `enableDamping`, `dampingFactor={0.05}`, `minDistance={0.5}`, `maxDistance={50}`. No polar angle clamps are set (full vertical orbit allowed).

## Phase 2 â€” Scene Graph Rendering (Active)

Phase 2 transforms a world from "one .glb file" to "a scene graph that composes many `.glb` assets in 3D space." The renderer split (sub-slice 8.1) is now shipped.

### Scene graph schema (v1)

The TypeScript schema lives in `src/lib/scene-graph/schema.ts`. Key shapes:

- `ObjectSchema` â€” `{ id, assetId (uuid FK to world_assets), name?, position, rotation (Euler radians), scale }`. Rotations are Euler vec3 in v1 for editor simplicity; planned switch to quaternion in v2 when an animation system needs the precision.
- `LightSchema` â€” discriminated union on `type`: `"ambient"` (`intensity`, `color`) or `"sun"` (`intensity`, `direction`, `color`). No point/spot lights in v1.
- `EnvironmentSchema` â€” `skybox` enum (`"studio" | "sunset" | "dawn" | "night" | "warehouse" | "park" | "city" | "forest"`) + nullable `fog` (`color`, `near`, `far`).
- `CameraSchema` â€” `position`, `target`, `fov`. In the renderer, `target` is intentionally not applied â€” `<Bounds>` takes over at mount. Revisit in v2 if authored target should override Bounds.
- Top-level `SceneGraphV1` â€” `{ schemaVersion: 1, objects[], lights[], environment, spawnPoints[], camera }`.

Parse with `parseSceneGraph(unknown): SceneGraphAny` â€” throws on invalid input; caller decides whether to fall through to legacy. Build an empty graph with `emptySceneGraph(): SceneGraphV1`.

### Renderer branching

- If `worlds.scene_graph` is present and parseable â†’ `<SceneGraphRendererClient>` â†’ `SceneGraphRenderer`
- Else (null or parse failure) â†’ `<WorldViewerClient>` â†’ legacy `WorldViewer`
- Branch point: `src/app/world/[id]/page.tsx` (task 8 / frontend-dev)

### `<SceneGraphRenderer>` â€” how it works

**Component structure:**

- `SceneGraphRenderer` (exported default) â€” outer shell: `WorldViewerErrorBoundary`, container div, `FullscreenButton`, `Canvas`, `LoadingOverlay`. Camera position + fov taken from `sceneGraph.camera`.
- `SceneGraphScene` â€” extracted to `SceneGraphScene.tsx` (named export); renders lights (discriminated union), `<Environment>`, `<fog>`, and one `<AssetObject>` per `sceneGraph.objects` entry. Extracted so it can be embedded in both `SceneGraphRenderer` and `WorldVisitor` (different Canvas wrappers, same scene content).
- `AssetObject` â€” named export in `SceneGraphScene.tsx`; calls `useGLTF(assetUrl)` and returns `<group position rotation scale><primitive object={cloned} /></group>`. The `cloned` is `useMemo(() => scene.clone(true), [scene])`.
- `Asset` interface â€” named export in `SceneGraphScene.tsx`; shared between `SceneGraphRenderer` and any other Canvas wrapper that embeds `SceneGraphScene`.
- `LoadingOverlay` â€” reads `useProgress().active` outside the Canvas (same pattern as `WorldViewer`).
- `FullscreenButton` â€” identical logic to the inlined version in `WorldViewer`.

**`scene.clone(true)` per instance:** drei caches `useGLTF` results by URL â€” all consumers of the same URL share one `THREE.Object3D` tree. Without cloning, two objects at different positions pointing at the same asset would fight over a single transform. `scene.clone(true)` gives each `<AssetObject>` its own independent copy of the tree. The binary is not re-fetched â€” the URL cache is still hit. Note: `scene.clone(true)` does NOT correctly deep-clone `SkinnedMesh` skeletons. Phase 2 v1 does not support skinned animations, so plain clone suffices. Switch to `SkeletonUtils.clone` if/when animation support arrives.

**Missing assets (non-fatal skip):** if `obj.assetId` is not found in the `assets[]` prop, a `console.warn` is emitted and that object is skipped. The rest of the scene renders normally. Load failures are never fatal.

**Lights rendering pattern:**
```tsx
{sceneGraph.lights.map((light, i) => {
  if (light.type === "ambient") {
    return <ambientLight key={i} intensity={light.intensity} color={light.color} />;
  }
  if (light.type === "sun") {
    return <directionalLight key={i} intensity={light.intensity} position={light.direction} color={light.color} />;
  }
  return null; // future light types
})}
```

**Environment + fog:**
```tsx
{sceneGraph.environment.skybox && <Environment preset={sceneGraph.environment.skybox} />}
{sceneGraph.environment.fog && (
  <fog attach="fog" args={[fog.color, fog.near, fog.far]} />
)}
```

**Spawn points** are not visualized in the viewer. They are stored in the scene graph for Phase 3 avatar entry only.

**Lazy-load pattern:** `SceneGraphRendererClient` uses `next/dynamic({ ssr: false })` to keep Three.js out of server bundles â€” same pattern as `WorldViewerClient`.

### Performance considerations

- **Instancing** â€” same `assetId` repeated many times should share GPU geometry (`<Instances>` from drei). Not implemented in v1; add during 8.4 stress-test if needed.
- **Frustum culling** â€” Three.js does this by default; ensure it's not disabled.
- **LOD** â€” not in v1; asset model designed to allow LOD variants later.
- **Async loading** â€” partial scene renders as individual GLB assets stream in (each `<AssetObject>` suspends independently within the shared `<Suspense>` boundary).

### Visitor mode (Slice 9.1 Chunks 1â€“5 + Slice 9.3 Chunk 5 â€” SHIPPED)

The visitor experience lives at `/world/[id]` for scene-graph worlds. It is a **separate rendering stack** from `SceneGraphRenderer` â€” they share `SceneGraphScene` (same scene content) but have a completely different camera/controls model.

**Two modes:**

- `"preview"` â€” default on page load. OrbitControls + the authored camera position (`sceneGraph.camera`). No pointer lock. Shows the "Enter world" CTA overlay.
- `"walking"` â€” first-person walk. Desktop: PointerLockControls + WASD + mouse-look. Touch: virtual joysticks + manual camera rotation. ESC or Exit button returns to preview.

**Component structure:**

- `WorldVisitorClient` (SSR-off wrapper) â†’ `WorldVisitor` (mode state, Canvas, overlays, touch detection, joystick ref) â†’ inside Canvas: `SceneGraphScene` (shared scene content) + `PresenceLayer` (always mounted) + either `PreviewMode` or `WalkMode`. Outside Canvas (DOM): `MobileJoysticks` (touch only, walking only) + `ControlsHint` (walking only).

**Desktop vs touch split (Chunk 5):**

`WorldVisitor` calls `useTouchDevice()` once at the top. The result (`isTouchDevice`) is passed as a prop into `WalkMode` and used to gate the `MobileJoysticks` render. The split:

- Desktop: `WalkMode` mounts `<PointerLockControls>`, registers keyboard listeners, listens for `pointerlockchange`. Mouse-look handled automatically by PointerLockControls. `useFrame` only applies `positionDelta` to `camera.position`.
- Touch: `WalkMode` returns `null` (no PointerLockControls), does NOT register keyboard or pointerlockchange listeners, does NOT call `.lock()`. `useFrame` reads `joystickInputRef`, maps left stick to `forward/strafe` (left.y inverted), right stick to `yawRate/pitchRate` (right.y inverted), calls `computeMovement`, then applies both `positionDelta` AND manually applies `yawDelta + pitchDelta` to camera rotation via `camera.rotation.set()` (YXZ order). Pitch is clamped to `Â±(Ď€/2 - 0.05)`.

**Joystick ref ownership:**

`WorldVisitor` owns `joystickInputRef = useRef<JoystickInput>({ leftX, leftY, rightX, rightY })`. `MobileJoysticks` writes to it via `handleLeftStick` / `handleRightStick` callbacks created in `WorldVisitor`. `WalkMode` reads the ref every frame (no re-renders). This avoids threading the ref through the Canvas boundary and keeps both components in sync on the same object.

**Mode switching and pointer lock:**

The "Enter world" button calls `setMode("walking")`, which mounts `WalkMode`. On desktop, `WalkMode`'s mount `useEffect` immediately calls `controlsRef.current.lock()`. This works because the React state flush from the button's onClick is synchronous â€” the gesture window is still open when `lock()` fires. If a browser ever tightens this, the fallback is to pass a callback from `WalkMode` to `WorldVisitor` and invoke `lock()` directly from the button's `onClick` handler (bypassing the render cycle). Documented in `WalkMode.tsx`. On touch, no pointer lock is requested.

**Walk mode exit:**

Desktop: two paths back to preview: (a) ESC â€” browser releases pointer lock, fires `pointerlockchange`, WalkMode's listener calls `onExit()`. (b) The "Exit" button in the top-right corner â€” calls `document.exitPointerLock()` which triggers the same `pointerlockchange` path.

Touch: `pointerlockchange` never fires (no lock was set). The Exit button calls `setMode("preview")` directly in `WorldVisitor`. WalkMode's `pointerlockchange` listener is skipped entirely (`!isTouchDevice` guard). The Exit button has a slightly larger tap target on touch (`px-4 py-3` vs `px-3 py-2`).

**Controls hint (Chunk 4/5):**

`ControlsHint` renders in walking mode on both platforms. `isTouchDevice` selects the copy: desktop shows WASD/mouse/Shift/ESC; touch shows stick/exit hints. Auto-dismisses after 12 seconds; persists to `localStorage` so it only shows once. z-index 60 (above joysticks at 50).

**Spawn point resolution:**

`WorldVisitor` reads `sceneGraph.spawnPoints`, prefers `id === "default"`, falls back to `spawnPoints[0]`, and finally falls back to `[0, 0, 5]`. Computed once via `useMemo` on mount.

**Canvas initial camera:**

Set to `[spawn.x, spawn.y + EYE_HEIGHT, spawn.z]` so both modes start in a sensible position. `PreviewMode`'s `useEffect` immediately repositions to `sceneGraph.camera.position`. `WalkMode`'s `useEffect` does the same with the spawn point.

**FullscreenButton and LoadingOverlay duplication:**

`SceneGraphRenderer.tsx` defines `FullscreenButton` and `LoadingOverlay` as private function declarations (not exported). `WorldVisitor.tsx` contains identical copies. If these are ever hoisted into a shared file, remove the duplicates and import from there.

**EYE_HEIGHT constant:**

`EYE_HEIGHT = 1.6` is exported from `WalkMode.tsx` and imported in `WorldVisitor.tsx` for the initial camera position calculation.

### Multi-user presence layer (Slice 9.3 Chunk 5 â€” SHIPPED)

Other visitors are rendered as capsule avatars inside the R3F Canvas. The implementation has two parts:

**`PresenceLayer` (Canvas-interior):**

- Always mounted regardless of mode (preview AND walking) â€” you see other visitors moving even before clicking "Enter world."
- Calls `useOthers()` from `@liveblocks/react`. Filters to only render users where `presence.inWalkMode === true` AND `presence.position !== null`. Users still on the preview screen are connected to the room but not rendered (presence position is `null` until they enter walk mode).
- `other.info` is accessed **directly** as `other.info` (Liveblocks v3 shape) â€” NOT `other.user.info`. This is `VisitorUserInfo` set server-side at JWT issue time via `prepareSession({ userInfo })`.
- No position lerp smoothing in v1. Liveblocks throttles presence at ~10 Hz; raw position jumps are acceptable. A lerp pass can be added later.
- Known v1 limitation: all users who enter walk mode spawn at the same point and their avatars overlap until they move.

**`VisitorAvatar` (pure presentation):**

- One `<group>` per remote visitor. No Liveblocks hooks â€” pure props.
- Capsule body: `<capsuleGeometry args={[0.3, 1.0, 4, 8]} />` â€” args are `(radius, height, capSegments, radialSegments)` per Three.js r168+ constructor. Total visual height = 1.0 + 2Ă—0.3 = 1.6 m (matches a humanoid). Capsule center is offset `y = -0.3` relative to the group so the body sits below the other user's eye-level position (the group's Y origin).
- Name tag: `<Billboard>` from drei + `<Text>` from drei (troika-three-text under the hood). Billboard auto-rotates to face the local camera regardless of the avatar's yaw. `<Text>` uses `outlineWidth` + `outlineColor` for readability on bright backgrounds.
- `userData = { collidable: false }` on the root group. The `collision.ts` wall-slide raycaster skips any mesh where `userData.collidable !== false`, so avatars do not physically block your walk path. This is the same mechanism used for non-collidable decorative props.

**`WalkMode` Liveblocks integration:**

- Imports `useUpdateMyPresence` from `@liveblocks/react`. Hook is called at the top of the component.
- On mount (`useEffect([])`): calls `updateMyPresence({ inWalkMode: true })`. Cleanup: calls `updateMyPresence({ inWalkMode: false, position: null })` so the local user's avatar disappears from other users' screens when they exit walk mode.
- Per-frame presence push (inside `useFrame`, after `camera.position.copy(safe)`): throttled to ~100 ms via `lastPushedAtRef`. Only fires when the user is actually moving (inside the `positionDelta.lengthSq() !== 0` branch) â€” idle users do not spam presence updates.
- Throttle strategy: client-side 100 ms guard PLUS Liveblocks's own default ~100 ms coalesce. The client guard primarily saves CPU (avoids JSON-serializing a position tuple at 60 Hz).

**TypeScript note:** `other.presence` from `useOthers()` comes back typed as `JsonObject` (Liveblocks's constraint type), not `VisitorPresence`, even with the global `Liveblocks` augmentation. The tuple `[number, number, number]` cannot be recovered from `JsonArray` by the constraint system. `PresenceLayer.tsx` uses `as unknown as VisitorPresence` (and similarly for `info`) â€” safe because `WalkMode` always writes exactly `VisitorPresence`-shaped data.

**Liveblocks provider wiring:**

`LiveblocksRoomProvider` (from Chunk 1/2) wraps `WorldVisitorClient` in `src/app/world/[id]/page.tsx` for scene-graph worlds only. Legacy GLB worlds get neither `LiveblocksRoomProvider` nor `PresenceLayer` â€” they don't support walk mode so presence is irrelevant.

### Editor mode (Phase 2, sub-slice 8.4 â€” SHIPPED)

The in-browser editor lives at `/world/[id]/edit`. It is a **separate rendering stack** from `SceneGraphRenderer` â€” they share the schema/asset-loading patterns but NOT the camera or interaction model.

**Components:**

- `src/components/editor/panels/Viewport.tsx` â€” the editor viewport (client component). Owns the `<Canvas>`, lighting, environment, floor grid, transform gizmos, click-to-select, click-to-deselect, orbit controls, and the Delete key shortcut.
- `src/components/editor/EditorAssetMesh.tsx` â€” renders one scene-graph object. Loads the asset GLB via `useGLTF`, clones the scene, applies position/rotation/scale, fires `selectObject()` on click, shows `<Outlines>` highlight when selected, registers its `Group` ref in the viewport's shared ref map.

**Key design choices:**

- **No `<Bounds>` in the editor** â€” `<Bounds>` auto-fits the camera on every scene change, which is wrong for an editor (camera would jump every time you add/move an object). The editor uses manual `<OrbitControls>` only.
- **Ref registration pattern** â€” `Viewport.tsx` holds a `Map<string, Group>` in a `useRef`. Each `EditorAssetMesh` calls `onRefChange(id, groupRef)` on mount and `onRefChange(id, null)` on unmount. When `selectedObjectId` changes, the viewport reads the map and passes the matching `Group` to `<TransformControls object={...}>`. No scene-tree traversal needed.
- **One op per drag** â€” `TransformControls` fires `onMouseDown` (disables OrbitControls) and `onMouseUp` (re-enables OrbitControls + reads final position/rotation/scale from the group ref + dispatches one `updateObject()` call). This creates exactly one operation per drag â€” correct for the undo stack and autosave.
- **OrbitControls disabled during drag** â€” while the gizmo is being dragged, `orbitRef.current.enabled = false` prevents camera pan from fighting the gizmo. Re-enabled on `onMouseUp`.
- **Selection highlight** â€” `<Outlines>` from drei (`thickness={0.025}`, `color="#22d3ee"` cyan-400). Falls back to a red wireframe cube for missing assets.
- **Floor grid** â€” `<Grid>` from drei, `infiniteGrid`, section size 5, cell size 1, zinc-700/zinc-600 colors, fade at 40 units.
- **Click-to-deselect** â€” a large invisible `<mesh>` (200Ă—200 plane at y=-0.002) catches raycasts on empty space. Object meshes call `e.stopPropagation()` so this only fires on miss.
- **Delete shortcut** â€” `useEffect` in `Viewport.tsx` handles Delete/Backspace when `selectedObjectId !== null` and focus is not in an input/textarea/contenteditable. Other shortcuts (T/R/S/Escape/Cmd+Z) remain in `EditorTopBar.tsx`.

**Route-segment layout:** `src/app/world/[id]/edit/layout.tsx` replaces the root layout's header + footer so the editor occupies the full viewport (`h-screen overflow-hidden`) without the global nav competing for vertical space. The `ClerkProvider` is already established by the root HTML shell â€” the segment layout just strips the nav chrome.

**Lighting in the editor** â€” same discriminated-union rendering pattern as `SceneGraphRenderer` (reads from `sceneGraph.lights`).

**Missing-asset fallback** â€” red wireframe `<boxGeometry>` cube with a `console.warn`. The editor still renders the rest of the scene normally.

### Editor presence layer (Slice 10.1 Chunk 2 â€” SHIPPED)

When 2+ editors open `/world/[id]/edit` for the same world, they see each other's cursor positions and selection highlights in real time. This is a pure presence overlay â€” no CRDT or concurrent mutation. Save conflicts are still handled by the existing 409+rebase cycle.

**Room sharing:** The editor page joins the same `world:${worldId}` Liveblocks room as walk-mode visitors. The `EditorPresenceLayer` filters `useOthers()` by `mode === "editor"` â€” walking visitors are invisible in the editor viewport. `PresenceLayer` (visitor side) filters by `isWalkingVisitor()` which rejects editor presence naturally.

**Provider wiring:** `src/app/world/[id]/edit/page.tsx` wraps `<EditorShell>` in `<LiveblocksRoomProvider worldId={id} initialPresence={INITIAL_EDITOR_PRESENCE}>`. Only the happy-path render gets the provider; forbidden/legacy/error paths do not.

**Broadcasting â€” `useEditorPresence` hook:**

- File: `src/components/editor/use-editor-presence.ts`
- Must be called inside the R3F Canvas tree (uses `useFrame` + `useThree`).
- Per frame: reads `pointer` (NDC Vector2 from R3F) + `camera` + `scene` + store state (`selectedObjectId`, `gizmoMode`). Calls `computeEditorPresence()` â†’ dedup check â†’ throttled `updateMyPresence()`.
- Throttle: 100ms client-side guard (same as `WalkMode.tsx`). Liveblocks's own ~100ms coalesce is a second layer.
- Dedup: `lastSerializedRef` stores a lightweight string of the last-pushed presence; identical consecutive frames are skipped.
- On mount: stamps `mode: "editor"` (belt + suspenders â€” provider already sets `INITIAL_EDITOR_PRESENCE`).
- Pure helper extracted: `computeEditorPresence(args)` performs the raycast + collidable filter + returns `EditorPresence`. Testable without WebGL.
- Wired via `<EditorPresenceWiring />` â€” a zero-render wrapper component defined in `Viewport.tsx` that calls the hook and returns null. Pattern makes future conditional mounting clean.

**`computeEditorPresence` pure helper:**

```ts
export function computeEditorPresence(args: {
  pointer: { x: number; y: number };   // NDC [-1, 1]
  camera: Camera;
  scene: Scene;
  raycaster: Raycaster;
  selectedObjectId: string | null;
  gizmoMode: GizmoMode;
}): EditorPresence
```

- Calls `raycaster.setFromCamera(pointer, camera)` then `intersectObjects(scene.children, true)`.
- Filters hits to exclude `userData.collidable === false` (gizmo helpers, remote cursor spheres, VisitorAvatars, invisible deselect plane).
- Returns `cursorWorldPos: null` when no usable hit (pointer outside canvas, only non-collidable meshes, etc.).

**Rendering â€” `EditorPresenceLayer` + sub-components:**

`EditorPresenceLayer` (R3F Canvas component, must be inside Canvas):
- Calls `useOthers()`, casts presence with `as unknown as UserPresence | null` (same TypeScript quirk as `PresenceLayer.tsx`).
- Filters with `isEditor(presence)` â€” rejects visitors, nulls, missing info.
- For each qualifying remote editor: renders one `<group key={connectionId}>` containing optionally `<RemoteEditorCursor>` (if `cursorWorldPos !== null`) and optionally `<RemoteEditorSelectionOutline>` (if `selectedObjectId !== null`).

`RemoteEditorCursor`:
- File: `src/components/editor/RemoteEditorCursor.tsx`
- Props: `{ position: [n,n,n], color: string, name: string }`.
- Sphere: `<sphereGeometry args={[0.08, 16, 16]}>` + `<meshBasicMaterial>` (unlit; color reads clearly regardless of scene lighting). Radius 0.08 â€” visible but non-intrusive.
- Name tag: `<Billboard>` + `<Text>` (same drei pattern as `VisitorAvatar.tsx`). Floats at `[0, 0.18, 0]` relative to the sphere center. Font size 0.08.
- `userData.collidable: false` on the root group â€” prevents cursor sphere from appearing in other raycast queries (walk-mode collision + the local editor's own cursor raycast).

`RemoteEditorSelectionOutline`:
- File: `src/components/editor/RemoteEditorSelectionOutline.tsx`
- Props: `{ objectId: string, color: string }`.
- Approach: Box3 fallback (v1). Traverses `scene.children` looking for the object by `userData.objectId` (added to `EditorAssetMesh.tsx`'s outer group in this chunk). Computes world AABB via `Box3.setFromObject()`, renders a wireframe `<boxGeometry>` (1.02Ă— expanded, `depthTest={false}`, `opacity={0.6}`).
- Skips render if `target` not found (GLB not loaded yet) or box is empty (degenerate geometry).
- Visually distinct from self-selection: self-selection uses drei `<Outlines>` (cyan, solid surface shader); remote outlines are wireframe boxes in the remote editor's HSL color.
- Known v1 limitation: axis-aligned bounding box â€” rotated objects get a larger box than necessary. Oriented bounding box is a future polish item.

**`EditorAssetMesh` change:**

`src/components/editor/EditorAssetMesh.tsx` outer group now has `userData={{ objectId: object.id }}` on both the normal and missing-asset fallback paths. This is additive â€” existing behavior unchanged. The tag allows `RemoteEditorSelectionOutline` to locate asset groups via `scene.traverse()` without needing access to Viewport's private ref map.

**TypeScript cast pattern (matches PresenceLayer):**

```ts
const presence = other.presence as unknown as UserPresence | null;
const info = other.info as unknown as VisitorUserInfo | undefined;
```

Safe because the broadcasting side (`useEditorPresence`) always writes exactly `EditorPresence`-shaped data, and `isEditor()` narrows at the render call-site.

## Gotchas

- **`useGLTF` caches by URL** â€” if you re-upload to the same URL, the cached version may persist. Force a refresh by appending a versioned query param (`?v=2`).
- **`<Suspense>` is required** around `useGLTF` consumers in R3F â€” provide a fallback or the canvas crashes on first load.
- **Pointer events on `<Canvas>`** â€” can interfere with page scrolling. Make sure the canvas doesn't capture wheel events when not focused.
- **R3F + Next.js App Router** â€” `<Canvas>` MUST be in a client component (`"use client"` directive at the top). Server-rendering R3F crashes.
